第一章 系统体系结构的内涵及要点

1.0 系统体系结构概述

什么是系统体系结构

架构的一般理解：

软件架构 = 软件的架构

词典中“架构”的定义：

  • 设计和建造建筑物的艺术与科学
  • 设计及构造的方式和方法
  • 部件的有序安排与结构
  • 计算机系统的总体设计或结构（包括硬件和软件，尤其是微处理器内部结构）

架构的起源

  • 起源于建筑学
  • 两个核心因素：
    • 基本建筑模块（砖、瓦、灰、沙、石等）
    • 模块之间的连接关系（如何组合成整体）
  • 建筑设计三原则：坚固、实用、美观

计算机硬件系统的“架构”

  • 基本模块：控制器、运算器、存储器、I/O设备等
  • 连接关系：总线
  • 常见结构：冯·诺依曼结构、存储层次结构、并行处理、流水线、多核CPU

“架构”的共性

  • 构件：基本构成要素
  • 连接件：要素之间的关系
  • 拓扑结构：物理分布
  • 约束：限制条件
  • 质量：性能等

软件架构（SA）定义

CMU观点

目前尚无统一、被普遍接受的定义，软件架构仍是一个年轻领域。

Garlan & Shaw（1994）

SA = {components, connectors, constraints}

包括：

  • 系统组织方式
  • 全局控制结构
  • 通信、同步、数据访问协议
  • 功能分配
  • 物理分布
  • 扩展性与性能
  • 演化维度
  • 设计替代方案选择

IEEE 1471-2000

SA = {component, connector, environment, principle}

  • 系统的基本组织结构
  • 构件之间、构件与环境之间的关系
  • 指导设计与演化的原理

另一种表达

SA = {elements, form, rational}

  • elements：处理元素、数据元素、连接元素
  • form：属性（限制选择）+ 关系（限制拓扑）
  • rational：选择依据

简化理解

架构 = 构件 + 连接件 + 拓扑结构 + 约束 + 质量

为什么学习软件架构

历史演进

  • 1960s：程序 = 算法 + 数据结构
  • 1970s：程序 = 子程序 + 子程序
  • 1980s：程序 = 对象 + 对象
  • 1990s：程序 = 构件 + 连接件

原因

  • 软件规模越来越大，需求复杂且频繁变化
  • 质量要求越来越高
  • 全局结构设计比算法和数据结构更重要
  • 很多质量属性体现在架构层面

当代软件规模示例

  • Windows 10：约5000万行代码
  • 阿里巴巴双十一：峰值每秒17.18亿条实时计算

架构师职责

  • 构件化设计
  • 全局控制结构
  • 通信协议
  • 异步处理
  • 数据存取
  • 功能分配
  • 物理部署
  • 扩展性与性能
  • 演化管理
  • 方案选择

架构师

角色定位

  • 技术领导（可能是一个团队）
  • 了解开发流程
  • 精通业务领域
  • 精于设计
  • 编程高手
  • 优秀表达能力
  • 果断决策
  • 谈判专家

架构师不是教出来的，是实践出来的。

软件架构的目标与作用

目标

  • 将复杂系统划分为模块
  • 规范模块构成
  • 组织完整系统
  • 保证质量要求

两方面

  • 外向目标：满足最终用户需求
  • 内向目标：满足设计者需要，易于实现、维护、扩展

作用

  • 交流手段：连接设计者、用户等
  • 可复用、可传递的抽象模型
  • 关键决策的体现（性能与安全、可维护性与可靠性、当前与未来成本之间的权衡）

在生命周期中的作用

  • 初期产品，成本低
  • 支持各方交流
  • 审查备选方案
  • 参考经典风格设计
  • 文档化
  • 指导开发
  • 支持测试
  • 作为维护依据

与其他设计活动的比较

  • 起点模糊：需求尚未明确时就要设计
  • 高抽象层次：关注构件和子系统关系
  • 分布决策：多方参与
  • 贯穿全局：影响整个项目周期

成为架构师的关键视角

  • 全局观：从全局设计软件
  • 折中观：在矛盾需求间权衡
  • 交流观：公共媒介沟通
  • 复用观：利用已有资产

软件质量

定义

软件与明确和隐含需求相一致的程度

影响质量的因素

  • 正确性、完备性、性能、安全性
  • 健壮性/可用性、易用性
  • 可维护性、可移植性、重用性
  • 集成性、可测试性、兼容性
  • 经济性等

架构手段示例

质量属性 架构手段
性能 集群、缓存、异步
安全性 防火墙、入侵检测、加密、单一入口
易用性 统一风格界面、定制
重用性 构件、框架、产品线
健壮性 集群、灾备、故障转移
可修改性 功能分解、设计模式、分层、解耦
可测试性 一致错误处理
集成性 最小化接口复杂度、统一命名、遵循标准
移植性 语言选择、分层、遵循标准

架构只能为质量创造条件，不能保证一定获得。

软件架构的技术演化

发展历程

  • 1960s：算法 + 数据结构
  • 1970s：子程序 + 子程序
  • 1980s：对象 + 对象
  • 1990s：构件 + 连接件
  • 2000s：服务 + 服务总线
  • Now：服务集群 + 中间件

1.2 系统体系结构与中间件

中间件的由来

  • 1968年：IBM CICS（应用与系统服务分离）
  • 1990年：BELL Tuxedo（分布式交易事务控制）
  • 1994年：IBM MQ系列（消息中间件）
  • J2EE：应用服务器中间件成为核心

中间件的定义

中间件是一组程序，应用于分布式系统各应用中，屏蔽底层通信，提供公共服务，保障高可靠性、高可用性、高灵活性。

中间件的定位

  • 通道：保障应用信息可靠传递
  • 桥梁：达成商务构件互联互通
  • 框架：促成企业应用完整集成
  • 平台：实现分布式应用顺利部署

什么是/不是中间件

  • ✅ 是软件
  • ✅ 在应用软件和系统软件之间
  • ✅ 是一类软件
  • ❌ 不是开发工具（但包含开发、管理、部署）

中间件的作用

  • 屏蔽异构性（硬件、OS、数据库差异）
  • 实现互操作（跨平台、跨网络）
  • 共性凝练和复用

中间件的意义

  • 缩短开发周期
  • 节约成本
  • 降低运行成本和故障率
  • 改善决策
  • 支持系统集成
  • 减少维护
  • 提高质量

中间件的分类

  1. 应用服务类：ORB、事务交易中间件、J2EE应用服务器
  2. 应用集成类：消息中间件、EAI、ESB、适配器
  3. 业务架构类：业务流程、业务模型、业务规则

常见类型：数据访问、消息、交易、分布式对象、分布式构件、工作流、应用集成、门户等

1.3 系统体系结构与中间件设计过程

4+1视图模型（Kruchten）

视图 描述 常用UML图
用例视图 系统典型场景与功能 Use Case Diagram
逻辑视图 抽象概念与功能（类、对象、接口） Class, Communication, Sequence Diagram
开发视图 子系统、模块、文件、资源 Component, Package Diagram
进程视图 进程及其通信协作 Activity, Sequence Diagram
物理视图 部署与配置 Deployment Diagram

逻辑视图 + 开发视图 = 静态结构
进程视图 + 物理视图 = 动态结构

案例：在线书店系统

功能需求

  • 用户管理：注册、验证、删除、修改
  • 存货管理：订书、入库、库存修改、查询
  • 购物管理：购物、购物车、支付
  • 订单实现：处理订单、打包、配送、查询

非功能需求

  • 可扩展
  • 可靠性
  • 安全性
