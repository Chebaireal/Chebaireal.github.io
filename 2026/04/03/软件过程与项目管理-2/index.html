一、软件工程的本质：不同抽象层次之间的映射与转换

任何软件系统开发的共同本质在于：从现实空间的需求到计算机空间的软件代码之间的映射与转换。

  • 单步映射：直接从需求到代码
  • 多步映射：通过中间抽象层（需求 → 设计 → 代码）

问题：单步映射与多步映射的优缺点分别是什么？

软件工程通过严格的规范和管理手段来缩小映射偏差，以牺牲“时间”提高“质量”。

两个核心映射

  1. 概念映射
    问题空间的概念 → 计算机解空间的模型化概念

    • 例如：“学生” → Class Student

  2. 业务逻辑映射
    问题空间的处理逻辑 → 解空间的处理逻辑

    • 例如：计算平均分 → calculateAverageScore()

软件工程需要解决：

  • 设置哪些抽象层次？
  • 每层的概念与表达方式？
  • 层间如何映射？

映射过程的具体活动

活动 输入 → 输出
需求分析 现实需求 → 需求规约
软件设计 需求规约 → 设计规约
实现 设计规约 → 代码
验证/确认 各阶段产物的一致性检查

二、软件工程关注的目标

产品与过程的二相性

  • 产品：各抽象层次的产出物（用例图、类图、代码、测试报告等）
  • 过程：各抽象层次之间的映射与转换

必须将产品与过程结合起来考虑，不能忽略任何一方。

功能性需求

  • 完备性：支持用户所有需求功能
  • 正确性：在需求范围内正确执行
  • 健壮性：在异常情况下正常运行（容错、恢复）
  • 可靠性：给定时间条件下不发生故障

非功能性需求

  • 效率（时间-空间）
  • 可用性（易用、易学）
  • 可维护性（适应变化）
  • 可移植性
  • 清晰性（易读、易理解）
  • 安全性
  • 兼容性
  • 经济性（成本、时间、市场）
  • 商业质量（上市时间、收益、生命周期）

目标之间的折中（Trade-off）

不同类型软件侧重点不同：

  • 实时系统：可靠性、效率
  • 长生命周期软件：可移植性、可维护性

多个目标同时最优不现实，目标之间常相互冲突。

三、软件开发中的多角色

客户方（甲方）

  • 决策者（CXO）
  • 终端用户
  • 系统管理员

开发公司（乙方）

  • 决策者
  • 销售与市场人员
  • 咨询师、需求分析师
  • 架构师、设计师
  • 开发经理/项目经理、程序员
  • 维护人员

不同角色的关注点（部分示例）

角色 关注点
开发人员 易调试性、可修改性
开发经理 可测试性
销售人员 结构清晰性
顾客 功能性
终端用户 易用性
系统管理员 性能、稳定性、易安装
维护人员 可维护性

思考题：不同角色的关注点是否有重叠或冲突？

四、软件工程 = 最佳实践

软件复杂性、易变性、不可见性使得高深理论难以直接应用。
软件工程被视为实践的艺术：经验越丰富，成功率越高。

经典最佳实践示例

Boehm 七条原理（1983）

  1. 用分阶段的生命周期计划严格管理
  2. 坚持进行阶段评审
  3. 实行严格的产品控制
  4. 采用现代程序设计技术
  5. 结果应能清楚地审查
  6. 开发小组的人员应少而精
  7. 不断改进开发过程

IBM RUP 最佳实践

  • 迭代化开发
  • 需求管理
  • 基于构件的体系结构
  • 可视化软件建模
  • 持续质量验证
  • 控制软件变更

Scrum 敏捷实践

  • 产品计划会议（用户故事）
  • 冲刺计划会议（分解任务）
  • 每日站会
  • 冲刺评审会
  • 冲刺回顾会
  • 2-4周迭代，冲刺期间不允许需求变更
  • 5-9人自组织团队

⚠️ 别人的经验在没有实践之前，对你来说只是抽象的概念。

与客户沟通的最佳实践（示例）

  1. 识别沟通对象
  2. 确定沟通方式
  3. 确定共同目标与范围
  4. 评审并修改范围说明
  5. 讨论典型场景与功能
  6. 记录并细化
  7. 分配优先级
  8. 最终评审
  9. 双方签字

五、软件工程的四个核心理论概念

概念 含义 核心问题
分而治之 将复杂问题分解为独立子问题分别求解 如何分解才能更易理解、开发、维护？
复用 使用已有架构、框架、模块组装新系统 不要重复造轮子（DRY）
折中 在矛盾需求间作出合理取舍 如何调和矛盾？
演化 软件不断成长以适应变化 如何设计使软件具有适应变化的能力？

其他相关概念：概念与形式模型、抽象层次、一致性与完备性、效率。

总结

本章核心要点：

  1. 软件工程本质是不同抽象层次之间的映射与转换
  2. 同时关注产品与过程，并权衡多种质量目标
  3. 软件开发是多角色协作，视角不同需求不同
  4. 软件工程 = 最佳实践的积累与运用
  5. 四大核心理论：分治、复用、折中、演化

💡 个人思考：软件工程不是纯理论，而是经验科学。多实践、多犯错、多总结，才能真正理解这些思想。
