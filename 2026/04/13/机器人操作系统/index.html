一、具身智能与ROS概述

1.1 什么是具身智能？

具身智能（Embodied Intelligence）是基于物理身体实现感知、行动与认知深度融合的智能系统。

核心特征：智能体通过实体载体与动态物理环境进行实时交互，获取环境信息、分析问题、制定决策并执行物理动作，区别于传统脱离物理载体的虚拟AI系统。

具身的核心内涵：为智能赋予物理载体，让AI算法与数据摆脱纯数字层面的局限，依托实体硬件与物理世界建立交互闭环。

三种智能形态辨析：

类型 特点
离身智能 不依赖物理载体，仅在虚拟符号空间内完成信息处理与推理
具身智能 以感知-行动闭环为核心，依托物理身体与环境直接交互
反身智能 对自身内部状态、决策逻辑的监控与调整，具备初步自我反思能力

传统AI与具身AI对比：

维度 传统AI 具身AI
环境 静态、封闭的数字空间 动态、开放的物理世界
输入 单一/特定模态数据 多模态实时传感器数据流
输出 数字信号 可落地的物理动作
目标 完成模式识别、内容生成等数字任务 在复杂物理环境中完成具体实操任务
学习方式 基于大规模静态数据集训练 通过与环境实时交互、试错迭代学习

1.2 什么是ROS？

ROS（Robot Operating System）是开源的机器人软件框架。虽然名字中带有操作系统，但实际上它是一个中间件，或者更简单地看成是一个应用程序的开发库。

ROS的功能包括：

  • 硬件抽象
  • 底层设备控制
  • 常用功能的实现
  • 进程之间的消息传递
  • 功能包管理
  • 在多台计算机之间获取、构建、编写和运行代码的工具和库

ROS的三大要素：

  • Plumbing（管道）：核心是消息传递系统，通过发布/订阅模式管理分布式节点之间的通信细节
  • Tools（工具）：包括launch、rviz、Gazebo、rqt等，用于调试、可视化、绘图、日志记录和回放
  • Capabilities（能力）：ROS生态系统是机器人软件的仓库，从驱动程序到算法再到用户界面

1.3 ROS预备知识

知识领域 要求
操作系统 熟悉Linux操作系统，掌握基本命令行操作
编程语言 C++（C++11/14/17标准）和Python
软件开发 掌握软件开发技术和方法，建立软件开发思维模式，使用面向对象技术进行程序设计

1.4 ROS1 vs ROS2

特性 ROS1 ROS2
通信中间件 自定义TCPROS，需要Master节点 DDS，无需中心节点
实时性 不支持 支持实时控制
安全性 缺乏内置安全功能 内置安全功能（认证、加密、访问控制）
跨平台 主要支持Linux 支持Linux、Windows、macOS、RTOS
嵌入式支持 较弱 通过Micro-ROS支持MCU
多机器人 无标准方法 原生支持多机器人系统

ROS1的缺陷：

  • 无实时性
  • 嵌入式设备不友好
  • 对网络通信重依赖
  • 稳定性欠佳，ROS Master可能宕机
  • 无多机器人系统标准方法

1.5 DDS（数据分发服务）

DDS是OMG组织发布的分布式通信规范，采用发布-订阅模型，以中间件形式提供通信服务，并提供多种QoS策略。

DDS的核心概念：

  • 域（Domain）：虚拟通信网络，只有同一域中的实体才能相互发现和交换数据
  • 主题（Topic）：命名的数据通道，发布者和订阅者通过主题名称进行通信
  • 发布者（Publisher）：发送数据的应用程序
  • 订阅者（Subscriber）：接收数据的应用程序
  • QoS策略：控制通信行为（可靠性、持久性、延迟等）

常用DDS实现：

  • Fast DDS（eProxima，开源）
  • Cyclone DDS（Eclipse，开源）
  • Connext DDS（RTI，商业）

DDS应用领域：

  • 国防军工（宙斯盾舰、捕食者无人机）
  • 自动驾驶（多传感器融合）
  • 工业自动化

1.6 Micro-ROS

Micro-ROS将ROS2扩展到资源受限的嵌入式平台和微控制器（MCU）。

核心架构：

  • MCU侧：运行Micro XRCE-DDS Client，支持FreeRTOS、Zephyr、NuttX等RTOS
  • 主机侧：运行Micro XRCE-DDS Agent，作为客户端与标准DDS世界的桥梁
  • 传输方式：支持UDP、TCP、串口等

1.7 ROS版本与支持周期

ROS版本命名按英文字母顺序，每年5月23日（世界乌龟日）发布新版本。

版本 支持期限
普通版本 2年支持
LTS版本 5年支持
  • ROS1 Noetic Ninjemys是ROS1的最终版本，已于2025年停止维护
  • ROS2 Jazzy Jalisco（2024-2029）是当前最新的LTS版本
  • 目前越来越多的机器人、无人机甚至无人车都采用ROS作为开发平台

1.8 ROS-Industrial（ROS-I）

ROS-I将ROS生态带入工业自动化领域，核心目的：

  • 架起研究与工业的桥梁
  • 为不同品牌工业机器人提供标准化驱动接口
  • 将ROS的高级功能（感知、运动规划）与工业机器人的底层可靠性结合

二、C++现代特性准备

ROS2源码使用C++11及更高版本，需要掌握以下C++新特性：

2.1 自动类型推导（auto）

auto是一种语法糖，告诉编译器根据初始化表达式自动推导变量的类型。在编译阶段，编译器会将auto替换为推导出的实际类型。

不使用auto的写法：std::vector::iterator it = vec.begin();

使用auto的写法：auto it = vec.begin(); // 编译器自动推导为std::vector::iterator

2.2 智能指针

智能指针用于管理动态分配的内存，避免内存泄漏和空指针问题。

std::shared_ptr（共享指针）：

  • 记录指向同一个资源的指针数量
  • 当数量为0时自动释放内存
  • 使用std::make_shared创建可以一次分配对象和控制块内存，效率更高

创建方式：

  • std::shared_ptr sp1 = p; 或 std::shared_ptr sp1(new Widget());
  • auto sp1 = std::make_shared(); // 推荐方式

常用方法：

  • use_count()：返回引用计数
  • get()：返回指向的内存地址
  • reset()：释放指针

std::unique_ptr：独占型指针，不能被复制
std::weak_ptr：弱引用指针，配合shared_ptr使用

2.3 Lambda表达式（匿名函数）

语法格式：[capture_list] (parameters) -> return_type { function_body }

捕获方式：

  • [x]：按值捕获变量x，只读
  • [&x]：按引用捕获变量x，可修改
  • [=]：按值捕获所有外部变量
  • [&]：按引用捕获所有外部变量
  • [=, &x]：混合捕获，x按引用，其他按值

应用场景：

  • 简化函数对象的定义
  • 实现回调函数

2.4 函数包装器（std::function）

std::function是C++11引入的通用函数包装器，用于存储、传递和调用任何可调用对象（普通函数、Lambda表达式、函数对象、成员函数等）。

语法：std::function<返回值类型(参数类型1, 参数类型2, …)>

适用场景：

  • 存储和传递函数、Lambda表达式、函数对象
  • 实现回调机制
  • 处理复杂的函数绑定和调用

2.5 参数绑定（std::bind）

std::bind用于将可调用对象与其参数绑定，生成一个新的可调用对象。可以部分绑定参数、重新排列参数顺序，或者绑定成员函数到对象实例。

占位符：std::placeholders::_1、std::placeholders::_2等

std::function与std::bind的区别：

维度 std::function std::bind
功能重点 存储和传递可调用对象 绑定可调用对象的参数，创建新的可调用对象
使用场景 回调机制、事件处理 预先绑定部分参数或调整参数顺序
返回值 类模板，封装可调用对象 函数模板，返回新的可调用对象

2.6 异步操作（std::future）

std::future用于表示异步操作的结果，通常与std::async、std::promise或std::packaged_task一起使用。

常用方法：

  • get()：获取异步操作结果（阻塞直到结果可用）
  • valid()：检查是否关联了有效的共享状态
  • wait()：阻塞直到结果可用
  • wait_for()：阻塞一段时间等待结果
  • wait_until()：阻塞直到某个时间点

2.7 线程间通信（std::promise）

std::promise用于在多线程编程中存储一个值或异常，并通过std::future获取该值或异常。它提供了一种线程间通信的机制，允许一个线程设置值，另一个线程获取值。

核心功能：

  • set_value()：设置一个值
  • set_exception()：设置一个异常
  • get_future()：返回std::future对象

std::promise和std::future是一对一的通信机制。

2.8 面向对象技术

使用面向对象技术进行程序设计，例如使用Pimpl（Pointer to Implementation）惯用法隐藏实现细节：

头文件中声明接口类和一个指向实现类的指针，实现类中封装具体实现细节。这种设计可以：

  • 减少编译依赖
  • 隐藏实现细节
  • 提高编译速度

三、ROS2核心概念

3.1 工作空间（Workspace）

工作空间是一个包含多个功能包的顶层目录，是ROS2项目开发的根环境。

目录结构：

  • src/：存放所有功能包的源代码（用户手动创建）
  • build/：编译过程中生成的中间文件（由colcon自动生成）
  • install/：编译后安装的可执行文件、库、配置文件等（由colcon自动生成）
  • log/：编译日志和运行时日志（由colcon自动生成）

作用：

  • 组织代码：将多个相关的功能包集中管理
  • 隔离环境：不同项目使用独立的工作空间，避免依赖冲突
  • 统一构建：通过colcon build一次性编译工作空间内所有包
  • 环境配置：通过source install/setup.bash将当前工作空间加入ROS2环境

创建和构建工作空间：

创建工作空间：mkdir -p ~/ros2_ws/src

构建工作空间：cd ~/ros2_ws && colcon build –symlink-install

设置环境：source ~/ros2_ws/install/setup.bash

覆盖机制（Overlay）：

  • 底层（underlay）：系统安装的ROS2核心包（如/opt/ros/jazzy）
  • 上层（overlay）：用户创建的工作空间
  • 当source上层工作空间后，它会覆盖底层中同名的包

3.2 功能包（Package）

功能包是ROS2中组织软件的最小单元，可以包含一个或多个节点、库、配置文件、启动文件、数据文件等。

两种构建类型：

  • ament_cmake：用于C++代码，使用CMake构建系统
  • ament_python：用于纯Python代码，使用Python setuptools构建

核心配置文件：

  • package.xml：包的元数据和依赖列表（名称、版本、描述、作者、许可证、依赖关系）
  • CMakeLists.txt（C++包）或setup.py（Python包）：定义如何编译和安装包中的内容

C++包目录结构：

  • include/：头文件目录
  • src/：源代码目录
  • launch/：启动文件（可选）
  • config/：配置文件（可选）

Python包目录结构：

  • 包名目录/：Python模块目录
  • resource/：资源文件（可选）
  • test/：测试文件（可选）

常用命令：

  • 创建功能包：ros2 pkg create –build-type –dependencies –node-name
  • 查看所有已安装的包：ros2 pkg list
  • 查看包信息：ros2 pkg xml

3.3 节点（Node）

节点是ROS2中可执行程序的一个实例，是执行具体任务的独立单元。

节点特点：

  • 功能专一：每个节点只做一件事
  • 独立运行：节点可以单独启动、停止，一个节点的崩溃不会直接影响其他节点
  • 语言无关：节点可以用C++、Python等语言编写
  • 可复用：节点可以轻松用于多个机器人项目

节点的组成：

  • 发布者（Publisher）：向某个话题发送消息
  • 订阅者（Subscriber）：从某个话题接收消息并调用回调函数
  • 服务端/客户端（Service Server/Client）：提供或调用同步请求-响应服务
  • 动作服务器/客户端（Action Server/Client）：处理长时间任务并提供反馈
  • 定时器（Timer）：周期性地执行回调函数
  • 参数（Parameter）：可动态配置的变量

节点类型：

类型 特点 适用场景
标准节点（非可组合） 独立进程，隔离性好 节点间依赖少，崩溃不影响其他节点
可组合节点（组件） 多节点共享进程，零拷贝通信 节点间频繁通信，资源受限环境
生命周期节点（托管节点） 内置状态机，可控启动/关闭 需要精细生命周期管理的工业应用

生命周期节点状态：

  • Unconfigured（未配置）：节点刚被实例化，未分配资源
  • Inactive（非活跃）：节点已持有资源，但不收发数据
  • Active（活跃）：节点正常执行业务功能
  • Finalized（已终止）：终态，节点将被销毁

常用命令：

  • 运行节点：ros2 run
  • 列出正在运行的节点：ros2 node list
  • 查看节点详细信息：ros2 node info
  • 重映射节点名称：ros2 run –ros-args -r __node:=new_name

3.4 话题（Topic）

话题是节点间异步、单向的通信机制，基于发布-订阅模型。

特点：

  • 异步通信：发布者发送数据后不等待接收者
  • 一对多：一个发布者可以有多个订阅者
  • 多对多：多个发布者可以发布同一话题

QoS配置：

  • 可靠性：Reliable（确保送达）/ Best-Effort（尽力而为）
  • 持久性：Volatile（不保留历史）/ Transient Local（为后订阅者保留数据）
  • 历史记录：Keep Last（保留最近N条）/ Keep All（保留所有）

常用命令：

  • 查看话题列表：ros2 topic list
  • 查看话题消息类型：ros2 topic list -t
  • 打印话题消息：ros2 topic echo /topic_name
  • 手动发布消息：ros2 topic pub /topic_name msg_type “{data: value}”
  • 查看话题信息（含QoS）：ros2 topic info /topic_name -v

3.5 服务（Service）

服务是同步、双向的请求-响应通信机制。

特点：

  • 同步阻塞：客户端发送请求后等待响应
  • 一对一：一个请求对应一个响应
  • 适用于短时任务（查询状态、触发操作）

服务接口文件（.srv）结构：

  • 请求部分（—上方）：客户端发送的数据
  • 响应部分（—下方）：服务端返回的数据

常用命令：

  • 列出所有可用服务：ros2 service list
  • 查看服务信息：ros2 service info
  • 查看服务类型：ros2 service type
  • 查找指定类型的服务：ros2 service find
  • 手动调用服务：ros2 service call ““

3.6 动作（Action）

动作是用于长时间运行任务的通信机制，结合了话题和服务的特点。

三部分结构：

  • 目标（Goal）：客户端发送的任务请求
  • 反馈（Feedback）：服务端执行过程中的进度信息
  • 结果（Result）：任务完成后的最终结果

特点：

  • 异步通信：客户端发送目标后立即返回
  • 可取消：客户端可以在执行过程中取消任务
  • 可反馈：服务端可以周期性地发送进度反馈
  • 适用于导航、机械臂控制等耗时任务

常用命令：

  • 列出所有可用动作：ros2 action list
  • 查看动作信息：ros2 action info
  • 查看动作类型：ros2 action type
  • 发送动作目标：ros2 action send_goal ““ –feedback

3.7 参数（Parameter）

参数是节点的配置变量，每个参数由键（参数名）和值组成。

参数类型：

  • bool、int64、float64、string
  • byte[]、bool[]、int64[]、float64[]、string[]

特点：

  • 可动态修改：节点运行时可以实时调整参数
  • 可监控变化：节点可以注册回调函数，在参数改变时执行相应操作
  • 类型安全：每个参数有固定类型

参数回调机制：

  • 预设置回调：在参数实际更改前被调用，可以验证新值并决定是否接受更改
  • 参数事件回调：参数成功更改后触发事件，可以被其他节点订阅

常用命令：

  • 列出所有节点的参数：ros2 param list
  • 获取参数值：ros2 param get /node_name param_name
  • 设置参数值：ros2 param set /node_name param_name value
  • 保存参数到文件：ros2 param dump /node_name > file.yaml
  • 从文件加载参数：ros2 param load /node_name file.yaml

3.8 启动文件（Launch File）

启动文件用于一次性启动多个ROS节点，并配置参数。

文件格式：

  • Python（推荐）：*.launch.py
  • XML：*.launch.xml
  • YAML：*.launch.yaml

Python启动文件模板：

from launch import LaunchDescription
from launch_ros.actions import Node

def generate_launch_description():
node1 = Node(package=”pkg_name”, executable=”exec_name”, name=”node_name”)
node2 = Node(package=”pkg_name”, executable=”exec_name”, name=”node_name”)
return LaunchDescription([node1, node2])

Node参数说明：

  • package：节点所在的ROS2包名
  • executable：可执行文件名称
  • name：节点名称（可选）
  • namespace：命名空间
  • parameters：参数设置（YAML文件或字典）
  • remappings：话题重映射
  • output：输出方式（screen/log）
  • arguments：命令行参数
  • ros_arguments：ROS2特定参数

常用命令：

  • 启动launch文件：ros2 launch pkg_name launch_file_name.py

四、ROS2通信机制深入

4.1 发布-订阅模式实现

C++发布者核心实现：

  1. 创建发布者：使用create_publisher模板函数，指定消息类型、话题名称和QoS配置
  2. 创建定时器：使用create_wall_timer，结合std::bind将回调函数绑定到当前对象
  3. 发布消息：填充消息对象后，通过发布者指针调用publish

C++订阅者核心实现：

  1. 创建订阅者：使用create_subscription模板函数，指定消息类型、话题名称、QoS配置和回调函数
  2. 实现回调函数：处理接收到的消息

Python发布者核心实现：

  1. 创建发布者：在节点的__init__方法中使用self.create_publisher()
  2. 创建定时器：使用self.create_timer()，指定时间间隔和回调函数
  3. 发布消息：实例化消息对象，填充数据，调用publish()

Python订阅者核心实现：

  1. 创建订阅者：使用self.create_subscription()，指定消息类型、话题名称、回调函数和QoS配置
  2. 实现回调函数：当新消息到达时自动调用

int64 sum

经过编译后，会生成三个C++类：

  • 顶层类AddTwoInts：容器类，内部嵌套Request和Response
  • AddTwoInts::Request类：对应请求部分
  • AddTwoInts::Response类：对应响应部分

C++服务端要点：

  • create_service是模板函数，需指定服务类型
  • 回调函数使用std::bind绑定成员函数，占位符_1和_2分别代表请求和响应
  • 请求和响应通过智能指针传递，回调中直接修改response即可

C++客户端要点：

  • create_client创建客户端
  • wait_for_service等待服务可用
  • async_send_request异步发送请求，返回SharedFuture
  • 可通过回调函数或future.get()获取响应

4.3 动作通信模式实现

动作接口文件（.action）示例：

目标部分

int64 num

结果部分

int64 sum

反馈部分

float progress

C++动作服务器要点：

  • 使用rclcpp_action::create_server创建动作服务器
  • 需要三个回调函数：处理目标（handle_goal）、处理取消（handle_cancel）、处理接受（handle_accepted）
  • 在执行函数中通过goal_handle->publish_feedback()发布反馈
  • 通过goal_handle->succeed()/abort()/canceled()设置结果

C++动作客户端要点：

  • 使用rclcpp_action::create_client创建动作客户端
  • async_send_goal异步发送目标
  • 可设置goal_response_callback、feedback_callback、result_callback

4.4 三种通信机制对比

特性 话题（Topic） 服务（Service） 动作（Action）
通信模式 发布/订阅（异步） 请求/响应（同步） 目标/反馈/结果（异步长时）
通信方向 单向数据流 双向（一次请求，一次响应） 双向（可多次反馈）
阻塞特性 非阻塞 客户端阻塞等待响应 非阻塞，可取消
节点关系 一对多，多对多 一对一 一对一
典型应用 传感器数据、周期性指令 查询状态、触发短时任务 导航、机械臂控制
接口文件 .msg .srv .action

五、服务质量（QoS）与分布式通信

5.1 QoS策略

QoS（Quality of Service）是一套用于定义和衡量数据分发服务质量的参数的集合。

核心QoS策略：

策略 说明 可选值
可靠性（Reliability） 控制消息送达保证 Reliable（确保送达）/ Best-Effort（尽力而为）
持久性（Durability） 新订阅者能否收到历史消息 Volatile（不保留）/ Transient Local（保留）
历史记录（History） 存储多少消息 Keep Last（保留最近N条）/ Keep All（保留所有）
深度（Depth） 队列最大消息数 整数值
截止时间（Deadline） 消息送达时间要求 毫秒值
活跃度（Liveliness） 节点存活状态 Automatic（自动）/ Manual（手动）
生存时间（Lifespan） 消息有效时长 时长值
所有权（Ownership） 发布者独占性 Shared（共享）/ Exclusive（独占）

预定义QoS配置：

配置常量 历史 深度 可靠性 持久性 典型用途
sensor_data Keep Last 5 Best Effort Volatile 传感器数据
parameters Keep Last 1000 Reliable Volatile 节点参数
default Keep Last 10 Reliable Volatile 通用配置
services_default Keep Last 10 Reliable Volatile 服务通信

QoS兼容性原则：订阅者的QoS策略不高于发布者的QoS策略时，两者才能成功建立连接。

5.2 分布式通信

ROS2通过DDS的域ID机制实现分布式通信。

ROS_DOMAIN_ID：

  • 域是虚拟通信网络，只有同一域中的节点才能相互发现和交换数据
  • 默认域ID为0
  • 取值范围：0-255（实际限制）

设置域ID的方法：

  1. 环境变量（最常用）：export ROS_DOMAIN_ID=42
  2. 命令行临时指定：ROS_DOMAIN_ID=42 ros2 run my_package my_node
  3. 在代码中设置：options.context()->set_domain_id(42)
  4. 通过launch文件设置：SetEnvironmentVariable(‘ROS_DOMAIN_ID’, ‘42’)

六、机器人建模与仿真

6.1 URDF（统一机器人描述格式）

URDF是使用XML格式描述机器人结构的文件。

核心标签：

  • robot：根标签，包含机器人模型的所有定义
  • link：连杆标签，描述机器人的刚性部件（底盘、轮子、摄像头等）
  • joint：关节标签，描述两个连杆之间的连接关系

link标签的子标签：

  • visual：描述外观（形状、尺寸、颜色）
  • collision：碰撞属性（碰撞检测的依据）
  • inertial：惯性矩阵（力学仿真计算的基础）

joint标签的属性：

  • name：关节名称
  • type：关节类型（continuous、revolute、prismatic、fixed等）

joint标签的子标签：

  • parent：父级连杆
  • child：子级连杆
  • origin：偏移量和姿态
  • axis：围绕哪个关节轴运动

关节类型：

类型 说明
continuous 旋转关节，可无限旋转
revolute 旋转关节，有角度限制
prismatic 滑动关节，沿轴线移动
fixed 固定关节，不允许运动
floating 浮动关节，允许平移和旋转
planer 平面关节，在平面内运动

6.2 Xacro（XML宏语言）

Xacro是URDF的优化方案，提供了编程能力。

主要特性：

  • 变量声明：<xacro:property name=”radius” value=”0.1” />
  • 数学运算：${radius * 2}
  • 宏定义：<xacro:macro name=”wheel_func” params=”name flag”>
  • 宏调用：<xacro:wheel_func name=”left” flag=”1” />
  • 文件包含：<xacro:include filename=”base.xacro” />

优势：

  • 代码复用，避免重复
  • 支持数学运算，避免手动计算错误
  • 提高代码可读性和维护性

6.3 Gazebo仿真

Gazebo是3D动态模拟器，用于显示机器人模型并创建仿真环境。

Gazebo版本：

  • Gazebo Classic：旧版本，将于2025年停止支持
  • Gazebo Sim（原Ignition）：新版本，模块化架构

URDF集成Gazebo的额外配置：

  1. 添加collision标签：碰撞检测依据
  2. 添加inertial标签：惯性矩阵，用于力学计算
  3. 重新设置材质：使用Gazebo特定颜色标签

惯性矩阵计算公式：

  • 球体：I = 2/5 * m * r^2
  • 圆柱：Ixx = Iyy = m*(3r^2 + h^2)/12, Izz = mr^2/2
  • 立方体：Ixx = m*(h^2 + l^2)/12, Iyy = m*(w^2 + l^2)/12, Izz = m*(w^2 + h^2)/12

常用Gazebo插件：

  • gz-sim-physics-system：物理参数定义
  • gz-sim-diff-drive-system：差速驱动控制
  • gz-sim-sensors-system：传感器仿真

6.4 RViz2可视化

RViz2是ROS的三维可视化工具。

主要功能：

  • 显示机器人模型
  • 显示激光雷达数据
  • 显示点云数据
  • 显示相机图像
  • 显示坐标变换（TF）

七、坐标变换TF2

7.1 基本概念

TF2是ROS中用于管理和处理不同坐标系之间变换关系的核心库。

核心概念：

  • 坐标系（Frame）：三维空间中的参考系
  • 变换（Transform）：两个坐标系之间的相对位置和姿态关系
  • 变换树（Transform Tree）：所有坐标系及其变换关系组成的树形结构
  • 时间戳：所有变换关联时间戳，支持历史查询

7.2 坐标变换消息类型

TransformStamped：用于传输坐标系相对位置信息

  • header：头信息（时间戳、坐标系ID）
  • child_frame_id：子坐标系ID
  • transform：坐标信息（平移向量、四元数旋转）

PointStamped：用于传输某个坐标系内的坐标点信息

  • header：头信息
  • point：点坐标（x, y, z）

7.3 静态坐标变换

适用于两个坐标系之间的相对位置固定不变的场景（如雷达/摄像头相对于底盘）。

发布方式：

  • 编码方式：创建StaticTransformBroadcaster，组织并发布TransformStamped消息
  • 命令行方式：ros2 run tf2_ros static_transform_publisher –x X –y Y –z Z –roll roll –pitch pitch –yaw yaw –frame-id frame_id –child-frame-id child_frame_id

7.4 动态坐标变换

适用于两个坐标系之间的相对位置动态变化的场景（如机械臂关节、多车编队）。

实现要点：

  • 创建TransformBroadcaster
  • 订阅位姿话题，获取当前位置
  • 根据订阅到的位姿生成坐标帧并广播

7.5 坐标变换监听

核心类：

  • tf2_ros::Buffer：存储坐标系变换信息的缓存区
  • tf2_ros::TransformListener：订阅tf话题，将接收到的变换信息存储到Buffer中

常用函数：

  • lookup_transform：查找两个坐标系之间的变换关系
  • can_transform：等待变换可用
  • transform：将坐标点从一个坐标系转换到另一个坐标系

7.6 TF2工具

  • static_transform_publisher：广播静态坐标变换
  • tf2_monitor：打印坐标系的发布频率与网络延迟
  • tf2_echo：显示特定坐标系的变换关系
  • view_frames：将坐标系的变换关系生成PDF文档

八、机器人感知

8.1 相机

相机类型：

  • USB摄像头（UVC相机）：即插即用，使用usb_cam功能包驱动
  • 深度相机（RGB-D）：提供彩色图像和深度信息（Intel RealSense、Kinect等）

ROS2中的图像消息类型（sensor_msgs/Image）：

  • header：时间戳和坐标系
  • height/width：图像高度和宽度
  • encoding：编码格式（RGB、YUV等）
  • step：一行图像数据的字节数量
  • data：存储图像数据的数组

压缩图像消息类型（sensor_msgs/CompressedImage）：

  • format：压缩格式（jpeg、png、tiff）
  • data：压缩图像数据

8.2 摄像头标定

标定的目的：

  • 消除镜头畸变（径向畸变、切向畸变）
  • 确定相机内参（焦距、主点）
  • 建立像素点到真实世界点的映射关系

标定类型：

  • 内参标定：相机内部参数（焦距、主点、畸变系数）
  • 外参标定：相机与机器人坐标系之间的变换关系

标定工具：camera_calibration功能包

标定参数说明：

  • fx/fy：焦距（像素单位）
  • cx/cy：图像光心坐标
  • k1/k2/k3：径向畸变系数
  • t1/t2：切向畸变系数

8.3 激光雷达

ROS2中的激光雷达消息类型（sensor_msgs/LaserScan）：

  • angle_min/angle_max：扫描起始/结束角度（弧度）
  • angle_increment：相邻激光点角度间隔
  • range_min/range_max：最小/最大测距距离
  • ranges：距离数据数组（核心数据）
  • intensities：反射强度数据

8.4 点云数据

ROS2中的点云消息类型（sensor_msgs/PointCloud2）：

  • height/width：点云高度和宽度（无序点云height=1）
  • fields：每个点的字段定义（x、y、z、intensity等）
  • point_step：单个点占用的总字节数
  • data：存储点云数据的二进制数组

8.5 OpenCV集成

cv_bridge功能包：在ROS2图像消息和OpenCV图像格式（cv::Mat）之间进行转换。

转换方式：

  • toCvCopy：深拷贝，适合需要修改数据的场景
  • toCvShare：浅拷贝，适合只读场景

image_transport：图像传输扩展包，支持压缩传输，降低带宽占用。

九、机器人控制（ros2_control）

9.1 概述

ros2_control是ROS2中用于机器人实时控制的核心框架。

作用：

  • 硬件抽象与解耦
  • 控制器复用
  • 仿真到实机无缝迁移
  • 实时性支持

9.2 核心组件

组件 说明
硬件组件 抽象物理硬件（系统、传感器、执行器）
控制器管理器 核心调度中枢，管理控制器生命周期
资源管理器 动态加载硬件组件，管理接口

9.3 硬件接口类型

  • 状态接口（State Interface）：只读，表示传感器读数
  • 命令接口（Command Interface）：读写，用于向硬件下发命令（具有互斥性）

9.4 常用控制器

  • joint_state_broadcaster：读取关节状态并发布到/joint_states
  • diff_drive_controller：差速驱动控制器，接收/cmd_vel输出轮速指令

  • 激活控制器：ros2 control switch_controllers –activate

十、导航与建图（Nav2）

10.1 Nav2架构

Nav2是ROS2中用于移动机器人自主导航的模块化框架。

核心组件：

组件 功能
BT Navigator Server 核心调度节点，使用行为树协调各模块
Planner Server 全局路径规划（A*、Dijkstra）
Controller Server 局部路径规划与运动控制（DWA、TEB）
Recovery Server 恢复行为（旋转、后退、重新规划）
Smoother Server 路径平滑优化
Waypoint Follower 执行路径点序列

10.2 代价地图（Costmap）

代价地图是多层叠加的地图，包含：

  • Static Map Layer：SLAM构建的静态地图
  • Obstacle Map Layer：传感器感知的障碍物信息
  • Inflation Layer：障碍物膨胀层（设置警戒区）

两种代价地图：

  • Global Costmap：覆盖整个已知环境，用于全局规划
  • Local Costmap：以机器人中心的小范围地图，用于局部避障

10.3 SLAM Toolbox

用于2D SLAM建图。

两种模式：

  • 同步模式（sync）：每帧激光数据立即处理，CPU效率高，适合低速机器人
  • 异步模式（async）：数据放入队列异步处理，不丢帧，适合高速机器人

常用命令：

  • 启动异步建图：ros2 launch slam_toolbox online_async_launch.py
  • 保存地图：ros2 run nav2_map_server map_saver_cli -f map_name

10.4 AMCL定位

AMCL使用自适应蒙特卡洛定位方法，基于粒子滤波推算机器人位置。

粒子滤波流程：

  1. 初始化：随机分布粒子
  2. 预测：根据运动模型移动粒子
  3. 更新：根据传感器测量更新粒子权重
  4. 姿态估计：计算加权平均值
  5. 重采样：去除权重小的粒子，复制权重大的粒子

常用命令：

  • ros2 run nav2_map_server map_server：加载地图
  • ros2 run nav2_amcl amcl：启动定位节点

10.5 常用导航命令

  • 查看话题列表：ros2 topic list
  • 手动发布速度指令：ros2 topic pub /cmd_vel geometry_msgs/msg/Twist “{linear: {x: 0.5}, angular: {z: 0.0}}”
  • 设置初始位姿：通过Rviz的2D Pose Estimate按钮
  • 设置导航目标：通过Rviz的2D Nav Goal按钮

十一、行为树（Behavior Tree）

11.1 基本概念

行为树是一种用于机器人决策和控制的树状结构。

节点类型：

类型 说明
控制节点 定义子节点执行顺序（Sequence、Fallback、Parallel）
装饰节点 修改子节点行为（Inverter、ForceSuccess、Repeat）
条件节点 检查条件，返回Success/Failure
动作节点 执行具体任务，可返回Running

节点返回状态：

  • Success：成功
  • Failure：失败
  • Running：执行中

11.2 行为树 vs 有限状态机

维度 行为树 有限状态机
模块化 易于模块化，可复用子树 状态和转换耦合
故障恢复 内置恢复机制 需手动添加转换
可读性 层次化结构清晰 状态爆炸时难以维护
动态性 实时评估条件 需预定义所有转换

  • BT::SyncActionNode：同步动作节点（阻塞执行）
  • BT::AsyncActionNode：异步动作节点（非阻塞）
