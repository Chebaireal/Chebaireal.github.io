一、软件的基本概念

一个例子：从“程序”到“软件”到“服务”

阶段 形态 特点
给孩子用 程序 规模小，需求简单
给全校用 软件 规模扩大，需求复杂
放在网站上 服务 规模巨大，需求多变

规模越来越大，需求越来越复杂，难度越来越大，所需时间越来越长，出错的概率也越来越大

什么是软件？

软件（Software）：一组对象或项目所形成的一个**“配置”，由程序、文档和数据**等部分构成。

  • 程序（Program）：可被计算机硬件理解并执行的一组指令，提供期望的功能和性能
  • 数据（Data Structure）：程序能正常操纵信息的数据结构
  • 文档（Document）：与程序开发、维护和使用有关的图文材料

⚠️ 程序 ≠ 软件

软件的四大特征

特征 说明
复杂性（Complexity） 现实问题复杂，手工开发，智力密集型，与人的水平密切相关
不可见性（Invisibility） 未完成的软件看不见，难以刻画和度量，沟通困难
易变性（Changeability） 应用环境（人群、法规、硬件、领域）频繁快速变化
一致性（Conformity） 各子系统接口必须协同一致，随时间演变难以维持

软件为何需要不断变化？

  • 适应新的计算环境或新技术
  • 功能增强以实现新的业务需求
  • 与其他软件系统进行互操作
  • 重构以延续生命周期

软件不会磨损和老化，但维护困难

遗留系统（Legacy System）

  • 仍在使用中的软件系统，可满足客户需求，但难以“优雅地”演变
  • 据统计：**60%**的软件维护费用用于增加新功能，**17%**用于修正bug

软件为何无法被“制造”？

土木工程师 软件工程师
清楚知道精确坐标位置 需求可能做重大修改
车辆数年内不会重大改变 技术或业务频繁变化
按已验证的工艺施工 设计不到位就开工

软件的特性（幽默版）

  • Mechanical Engineering：在亮灯的房间找猫
  • Chemical Engineering：在黑灯的房间找黑猫
  • Software Engineering：在黑灯、没有猫的房间找黑猫
  • System Engineering：在黑灯、没有猫的房间找黑猫，还有人喊“我找到了”

软件的分类

按类型：

  • 系统软件：操作系统、编译器、数据库、IDE
  • 应用软件：商业软件（企业/政府）、个人软件、工程/科学计算软件

按开发方式：商业软件、开源软件

按存在形式：

  • 嵌入式软件/单机软件/分布式软件
  • 移动终端软件
  • 基于Web的软件
  • 服务形态的软件（SaaS）

二、软件的发展

软件开发方式的发展阶段

阶段 特点 编程语言/方法
1950-60年代 数值计算，依赖个人才能，出现“软件危机” Fortran
1970年代 商务领域推广，结构化编程，数据库，“软件生命周期”概念形成 Pascal, C
1980年代 规模扩大，CASE工具，质量度量，面向对象思想出现 C++, OO
1990年代 Internet/Web，分布式，软件复用，网格技术 Java, 构件化
2000年代至今 软件服务化，云计算，MDA，敏捷，SOA，普适计算 云平台，移动计算

软件开发技术的发展过程

年代 软件形态
1950-60 软件 = 程序 = 算法 + 数据结构
1970 软件 = 程序 + 文档 + 数据
1980-90 面向对象软件 = 对象 + 消息
1990s-今 面向构件软件 = 构件 + 框架
2000s-今 面向服务软件 = 服务 + 消息 + 总线

软件产业的发展阶段

  1. 企业定制独立的程序（大硬件公司下的部门）
  2. 早期小众软件产品（可重用，“拷贝”销售）
  3. 企业解决方案（SAP、Oracle、ERP）
  4. 大众应用软件包（Windows、Office、WPS）
  5. 网络软件与服务（以服务形式提供）

三、软件工程的基本概念

软件开发失败的例子

IBM OS/360操作系统（1963-66）：

  • 投资几千万美元
  • 工作量5000多人年
  • 拖延几年才交付
  • 每年发现近100个错误

F.Brooks：“正像一只逃亡的野兽落到泥潭中做垂死的挣扎，越是挣扎，陷得越深，最后无法逃脱灭顶的灾难。”

No Silver Bullet（没有银弹）- F. Brooks

核心观点：

  • 复杂的软件工程问题无法靠简单的答案来解决
  • 所有软件研发都包括essential task（本质任务）和accidental task（偶然任务）
  • 没有银弹 / 变化永恒 / 人月神话 / 协作沟通

软件危机（Software Crisis）

计算机软件的开发和维护过程中遇到的一系列严重问题：

表现
对开发成本和进度的估算很不准确，严重拖期和超预算
无法满足用户需求，用户很不满意
质量很不可靠，经常失效
难以更改、调试和增强
没有标准、完整的文档
软件成本严重上升
开发生产率跟不上应用发展

统计数字：混乱的软件开发

Standish Group 2004年调查（365家公司，8380个项目）：

  • 成功交付：34%
  • 非完美交付：53%
    • 预算：平均189%
    • 时间：平均222%
    • 功能：平均61%
  • 失败取消：13%

为何出现这种情况？

客观上：软件规模扩大，复杂度和难度呈指数级增长

主观上：

  • 缺乏工程性、系统性的方法论
  • 对软件开发的误区（软件神话）

软件神话（Software Myths）

角色 神话
客户 大概了解目标就够，可以逐步补充需求；外包后可以完全不管
管理者 有标准规程“宝典”就能解决一切；增加人手可赶上进度；用了先进工具就一帆风顺
开发者 客户很了解需求；交付后任务结束；运行后才能评估质量；程序是唯一交付成果；文档降低效率

四、软件工程的定义

“软件工程”的提出

  • 1968年，NATO在德国召开国际会议，首次提出“软件工程”概念
  • 背景：计算机应用发展 → 软件数量多规模大 → 软件成本高质量低 → 个体化开发 → 维护困难 → 软件危机

软件工程的定义

来源 定义
NATO 1968 为了经济地获得能够在实际机器上高效运行的可靠软件而建立和使用的一系列工程化原则
IEEE 将系统性的、规范化的、可定量的方法应用于软件的开发、运行和维护，即将工程化应用到软件上
国务院学位委员会 应用计算机科学理论和技术以及工程管理原则和方法，按预算和进度实现满足用户要求的软件产品的定义、开发、发布和维护的工程

归纳：软件工程是…

维度 内容
范围 过程、方法、工具、质量
目标 满足用户需求、按时交付、控制成本、高质量软件
本质 重要的不是技术，而是如何开发软件项目的思想；是一种建模活动、解决问题的活动、知识获取的活动

五、软件工程的知识体系

软件工程的主要范畴

1. 软件开发方法学

类型 核心 特点
结构化方法 程序 = 数据结构 + 算法 自顶向下，面向过程
面向对象方法 程序 = 对象 + 消息 自底向上，面向对象

2. 软件工具与软件工程环境

  • 软件工具：自动/半自动辅助开发（项目管理、需求管理、设计建模、编程调试、测试维护）
  • CASE工具：计算机辅助软件工程工具
  • 软件工程环境：多个工具和基础设施集成

3. 软件过程

  • 管理和控制产品质量的关键
  • 由一系列活动与步骤组成
  • 将人员、技术、组织与管理有机结合
  • 软件过程模型：瀑布、增量、迭代、敏捷等

4. 软件工程管理

目的：按进度和预算完成开发计划

内容：成本估算、进度安排、人员组织、质量保证

5. 软件质量特性

  • 软件产品与需求相一致的程度
  • Bug = 软件的行为和用户的期望值不一样

微软观点：完美的软件不存在，关键是决定什么时候“足够好”

SWEBOK（软件工程知识体系）

  • IEEE CS与ACM联合制定
  • 1994年开始研究
  • 2001年SWEBOK 1.0，2004年2.0，2014年3.0
  • 目的：定义学科边界，描述内容特征，为课程开发和职业认证提供依据

C-SWEBOK（中国软件工程教指委）

中国软件工程知识体系标准

六、软件工程工具（CASE工具）

类别 工具示例
制图、建模工具 Rational Rose, Microsoft Visio, StarUML
原型制作工具 Balsamiq Mockups, ExtJS, jQuery, EasyUI, 墨刀
集成开发环境 Visual Studio.NET, Eclipse
数据库设计工具 PowerDesigner, ERWin
代码生成工具 Rational Rose（含）
项目管理工具 Microsoft Project, 禅道, 领歌（Leangoo）
软件配置工具 Git, GitHub, GitLab, GitEE（码云）

总结

核心概念 要点
软件定义 程序 + 数据 + 文档
软件四大特征 复杂性、不可见性、易变性、一致性
软件危机 成本超支、进度拖期、质量低下、维护困难
软件神话 客户、管理者、开发者的各种误解
没有银弹 复杂问题没有简单答案
软件工程 系统性、规范化、可定量的工程化方法
知识体系 SWEBOK / C-SWEBOK
CASE工具 辅助软件开发全过程的计算机工具

💡 个人思考：从“程序”到“软件”到“服务”，规模越大，工程化的必要性越强。软件工程不是限制创造力的枷锁，而是保证复杂软件项目成功的必要方法论。没有银弹，意味着需要持续学习、实践和反思。
